## 1. Pasteboard capture veto — correctness & bypass vectors

**Verified correct (the core invariant holds).**
`Packages/GanchoKit/Sources/ClipboardCore/MacPasteboardMonitor.swift:195-240` runs, in
order and all metadata-only, *before* any content read: change-count gate → one-shot user
skip → self-write marker → `SensitivePasteboardTypes.captureVeto` (`:211`) → source-app
denylist (`:217-221`) → preference pre-filter → `scheduleRead` (`:236`). The veto set
(`SensitivePasteboardTypes.swift:17-18`) covers `org.nspasteboard`
Concealed/Transient/AutoGenerated. Pause semantics are right: content copied while paused
is *discarded without reading* (`:183-187`, `discardChangesWhilePaused` `:244-247`).
In-flight read coalescing (`:253-268`) prevents the stale-metadata/new-bytes mixup.

Residual bypass vectors (all inherent to the platform, none a code bug):

| # | Vector | Risk | Impact | Fix | Sev | Eff |
|---|---|---|---|---|---|---|
| 1.1 | **TOCTOU between veto and read.** `pollOnce` checks `types` at time T, `scheduleRead` reads content asynchronously at T+ε. If a password manager writes a concealed item in that window, the read returns the *secret's bytes* under the *previous change's* verdict. Partially mitigated: the change counter will tick again and the concealed item's own poll is vetoed — but the captured bytes are already the secret's if NSPasteboard returned the new content. | Low likelihood (window is ms), high impact (stored secret) | Secret persisted un-flagged | In `readDetached`, re-check `changeCount` *and* re-read `types` after the content read; discard if either moved or a veto type appeared (`MacPasteboardMonitor.swift:258-268`). Cheap, metadata-only, closes the race. | P2 | S |
| 1.2 | **Non-conforming apps.** Terminals, browsers, IDEs never set `org.nspasteboard` types; a password copied from a web page is captured. This is exactly why `SensitiveDataDetector` exists (defense-in-depth), so its false-negative surface (§4) *is* the veto's false-negative surface. | Medium | Secret stored (mitigated: detector + 10-min expiry) | Close the detector gaps (§4 — partially done in this pass). | P2 | S |
| 1.3 | **Detector is toggleable.** `ClipboardCore/IntelligencePreferences.swift:18` — secret detection is a user toggle ("the password-manager veto is separate and always on"). With the toggle off, layer 2 is gone entirely; layers 1 (veto) and denylist remain. Defensible product choice, but the off-state should be visible in the Privacy Center, not just Settings. | Low | User self-inflicted exposure | Surface "secret detection: OFF" as a Privacy Center warning row. | P3 | S |
| 1.4 | **iOS surfaces bypass nothing but rely on decorate-at-ingest.** `Apps/GanchoiOS/GanchoiOSApp.swift:946` and `Apps/GanchoShared/SharedCapture.swift:43` both run `SensitiveIngestionPolicy.decorate` at ingest — consistent with macOS (`Apps/GanchoMac/AppModel.swift:107`). Verified: all three capture funnels apply the detector. No change. | — | — | — | — | — |

## 2. SQLCipher key management & Keychain attributes

**File:** `Packages/GanchoKit/Sources/GanchoKit/KeychainPassphraseStore.swift`

Verified sound (confirms audit 01 F-3.6): 256-bit `SecRandomCopyBytes` key, hex-encoded
(`:153-160`), never derived from user input, `kSecAttrAccessibleAfterFirstUnlock` (`:139`) —
the most protective level compatible with both background capture and
`kSecAttrSynchronizable` (`:138`, iCloud Keychain replication; `…ThisDeviceOnly` is
incompatible). First-launch race is idempotent (`:68-82`). `Failure` carries only
`OSStatus` (`:26-32`). Passphrase is defensively SQL-escaped even though hex can't contain
quotes (`GRDBClipboardStore.swift:118-121`).

Deeper findings:

| # | Finding | Risk/Impact | Fix | Sev | Eff |
|---|---|---|---|---|---|
| 2.1 | **Unreadable-item recovery deletes the key** (`KeychainPassphraseStore.swift:121-127`): if `SecItemCopyMatching` returns data that fails UTF-8 decoding, the code *deletes the item* and regenerates. A transient decode anomaly would orphan the database permanently (new key ≠ old key → SQLCipher can't open → "reset the store" path → total history loss). The condition is near-impossible (we wrote UTF-8 hex), but the failure mode is maximal. | Very low likelihood, catastrophic impact | Before deleting, park the undecodable item under a renamed account (`gancho-sqlite.corrupt-<ts>`) instead of destroying it, so recovery remains possible. | P2 | S |
| 2.2 | **No `kSecUseDataProtectionKeychain` on macOS.** The macOS app + CLI use the legacy file-based login keychain path (`:20` "the unsandboxed `gancho` CLI share the user keychain without a group"). Legacy-keychain generic passwords are readable by any app the user approves via the ACL dialog — and users click "Always Allow". The data-protection keychain gives per-app-identity enforcement. Trade-off: it requires an entitlement/access-group story for the CLI (which is exactly the MCP auth work, §5). | Local-malware key theft after one user click | Adopt `kSecUseDataProtectionKeychain: true` + a shared access group for app+CLI when the MCP auth work lands. | P2 | M |
| 2.3 | **Key synchronizability is a deliberate, documented trade** (`:10-12`): iCloud Keychain holds the DB key, so Apple-account compromise ⇒ key compromise. `docs/ARCHITECTURE.md` says this honestly ("never zero-knowledge"). Keep the honesty; no code change. | — | — | — | — |
| 2.4 | **`PRAGMA cipher_memory_security` not set.** SQLCipher supports locking/zeroing its page cache; GRDB's `prepareDatabase { try db.usePassphrase() }` (`GRDBClipboardStore.swift:49-51`) doesn't enable it. Marginal (memory-resident secrets are out of scope for a clipboard manager whose *job* is holding the data), but cheap. | Cold-boot/memory-dump hardening | `try db.execute(sql: "PRAGMA cipher_memory_security = ON")` in `prepareDatabase`; measure the perf cost first (it is not free). | P3 | S |

## 3. Blob & thumbnail encryption at rest

**File:** `Packages/GanchoKit/Sources/GanchoKit/BlobStore.swift`

Verified sound: AES-GCM (authenticated) with magic header (`:195-211`), key derived from
the same Keychain passphrase (`:38-46` — 64-hex decodes to the raw 32 bytes; otherwise
SHA-256 of the trimmed string), thumbnails sealed too (`:99-107`), plaintext-thumbnail
file URLs refused for encrypted stores (`:112-113`, and the seal-safety contract at
`GRDBClipboardStore.swift:614-624`), one-time in-place migration with a marker
(`:144-152`).

| # | Finding | Risk/Impact | Fix | Sev | Eff |
|---|---|---|---|---|---|
| 3.1 | **Blob key == DB key (one key, two cryptosystems).** `BlobStore.encryptionKeyData(for:)` reuses the SQLCipher passphrase bytes as the AES-GCM key. Not exploitable today (both are secret, both AEAD-protected), but key-separation hygiene says derive per-purpose subkeys so a future weakness in one layer can't cross-contaminate. | Low | Derive with HKDF (`CryptoKit.HKDF`, info: `"gancho-blob-v1"`). Needs a migration path for existing sealed blobs (decrypt-reencrypt sweep like `encryptPlaintextFilesIfNeeded`), so **not** a mechanical change — do it Mac-side with tests. | P3 | M |
| 3.2 | **AES-GCM nonce handling is safe by construction** — `AES.GCM.seal` generates a random 96-bit nonce per call and `combined` carries it (`:198-202`). With content-addressed writes (`needsWrite` guard `:57-59`) the same key+nonce pair is never reused deliberately. Verified, no change. | — | — | — | — |
| 3.3 | **Orphan sweep can't leak** — `removeAll(except:)` (`:157-169`) deletes both blob and thumbnail; export writes *decrypted* blobs into the archive (`GanchoArchive.swift:76-80`), which is by design (portable format) but means the `.ganchoarchive` on disk is plaintext. Covered by §7 (export exposure); with the applied excludeSensitive default, secret payloads no longer reach it. | — | — | — | — |
| 3.4 | **Sensitive *blob* clips:** the detector only runs on text; a screenshot of a password is stored as a normal image (OCR exists — `ImageTextExtractor.swift` — but is not wired into the sensitive decision). | Screenshot-of-secret persists indefinitely | Feed extracted OCR text through `SensitiveDataDetector` when OCR is enabled; flag the clip (not the text) sensitive. | P2 | M |

## 4. SensitiveDataDetector — false-negative surface

Pre-fix coverage: cards (Luhn), AWS access/secret, Stripe `sk|rk_live|test_`, GitHub

`SensitiveDataDetector.Category` exists anywhere** — the only consumers are
`SensitiveIngestionPolicy.decorate`'s `finding != .creditCard` comparison
(`SensitiveDataDetector.swift:222` post-fix) and the test suite). Eight categories added
(`Category` enum `:9-26`), each with positive tests and new negatives in

| Added category | Pattern (post-fix line) | Notes |

| `openAIKey` | `\bsk-(proj-\|svcacct-\|admin-)?[A-Za-z0-9_-]{20,}\b` (`:70`) | Disambiguated from Stripe (`sk_` underscore vs `sk-` hyphen); `sk-learn` prose stays clean (negative test). |

Remaining known false negatives (deliberately **not** implemented — each has a real FP cost
that needs product judgment and on-device dogfooding):

| # | Missing pattern | Why deferred | Sev | Eff |
|---|---|---|---|---|
| 4.1 | **Bare JWT as secret** (`eyJ…\.…\.…`) | Gancho ships a JWT *decoder* Dev Action — developers copy JWTs constantly, most of them worthless test tokens. Ready pattern: `\beyJ[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{16,}\b`. | P3 | S |
| 4.2 | GitHub fine-grained PATs (`github_pat_[0-9A-Za-z_]{82}`) | Zero FP risk; simply wasn't in scope of this pass's mandated list. One regex + one case + tests. | P3 | S |
| 4.3 | Twilio (`SK[0-9a-f]{32}`), SendGrid (`SG\.[\w-]{22}\.[\w-]{43}`), Anthropic (`sk-ant-…` — already covered by the new openAIKey rule's prefix match), private SSH `PuTTY-User-Key-File` | Long tail; add opportunistically with the same case+regex+test recipe. | P3 | S |
| 4.4 | **Structural gap: detector sees only whole-text.** A secret inside a >N-char clip is still scanned (patterns are `CONTAINED`, good), but there's no cap — a 10 MB text clip runs ~15 regexes over the whole thing on the capture path. | Perf, not security | Cap the scan at the first ~64 KB; secrets are near the top of real pastes. | P3 | S |

## 5. MCP local server — auth/authorization threat model

**Files:** `GanchoKit/MCPAccess.swift`, `GanchoMCP/MCPServer.swift`,
`GanchoMCP/MCPToolRunner.swift`, `GanchoMCP/MCPStdioTransport.swift`,
`gancho/GanchoCLI.swift:168-183` (`runMCP`).

What holds (verified): off-by-default with fail-safe load (`MCPAccess.swift:71-79` —
missing/corrupt config ⇒ OFF/metadata); the sensitive veto in *every* scope
(`MCPToolRunner.swift:66,80-84,105-108,140`); scope gates on content, not verbs; every
call logged content-free (`MCPAccessEvent`, `:33-52`); disabled server completes the
handshake but advertises zero tools and refuses calls (`MCPServer.swift:9-11,74-78`);
binary payloads are described, never dumped (`MCPToolRunner.swift:157-165`).

The threat model gap (extends audit 01 F-3.4 with specifics):

| # | Finding | Risk/Impact | Fix | Sev | Eff |
|---|---|---|---|---|---|
| 5.1 | **Authorization is a world-writable-by-owner JSON file.** `mcp-config.json` in the store dir (`MCPAccess.swift:58-88`) is the *entire* grant: any process running as the user can write `{"isEnabled":true,"scope":"all"}` and then spawn `gancho mcp` — or skip MCP entirely and read `gancho.sqlite`… except the DB is SQLCipher-encrypted, so the *file* isn't enough; the attacker needs the Keychain key, which triggers a legacy-keychain ACL prompt (§2.2). So the honest chain is: malicious local process → silently flip config → spawn `gancho mcp` → *one keychain prompt in the CLI's name* → full non-sensitive history. The single human-visible step is that keychain dialog — and it names `gancho`, a binary the user trusts. | Malicious local process reads full non-sensitive history with one misleading prompt | (a) Store an app-minted random capability token alongside the scope (in the *Keychain*, not the JSON); `gancho mcp` must present it, so raising scope requires the app UI, not a file write. (b) Log + Privacy-Center-notify on any scope raise observed at server start (diff last-seen scope). (c) §2.2's data-protection keychain makes the prompt story real. | P1 | M |
| 5.2 | **No per-client identity.** MCP over stdio means the *spawner* is the client; Claude Desktop, Cursor, and a malicious agent all look identical in the access log (no client name/PID recorded). | Audit log can't attribute access | Record `initialize`'s `clientInfo` (name/version — already parsed at `MCPServer.swift:49-50` vicinity) into `MCPAccessEvent`; still spoofable, but makes the Privacy Center log meaningful for honest clients. | P2 | S |
| 5.3 | **Input validation is solid.** Unknown tools/malformed args → in-band `isError` results, not crashes (`MCPToolRunner.swift:36-58`); limits clamped (`:63` — 1…100); UUIDs parsed, never interpolated into SQL (GRDB bindings throughout). Malformed JSON-RPC lines → parse error with null id (`MCPStdioTransport.swift:31-38`). Oversized-line DoS is bounded by `bytes.lines` buffering — worst case the CLI process (not the app) balloons. No change. | — | — | — | — |
| 5.4 | **Regex search over MCP = user-grade regex engine exposure.** `search_clips` accepts `mode: "regex"` (`MCPToolRunner.swift:190-196` → `GRDBClipboardStore.search` regex path, audit 01 F-2.4). `NSRegularExpression` (ICU) is not ReDoS-immune; an agent-supplied pathological pattern pins a CPU core on the O(N) scan. Contained to the CLI process, but it holds the DB read lock intermittently. | Local DoS by a permitted-but-hostile agent | Time-box the regex scan (e.g. 2 s deadline checked per batch) or restrict MCP search to exact/fuzzy. | P3 | S |

## 7. Export data exposure (JSON / CSV / .ganchoarchive)

**Files:** `GRDBClipboardStore.swift:626-690`, `GanchoArchive.swift`,
`Apps/GanchoMac/SettingsView.swift:195-208`, `Apps/GanchoiOS/GanchoiOSApp.swift:771-790`.

| # | Finding | Status |
|---|---|---|
| 7.1 | **CSV formula injection** (audit 01 F-3.3): `csvEscape` (pre-fix `:661-666`) wrote `= + - @`-leading fields verbatim; clipboard content is attacker-influenced by construction. **Fixed in this pass:** any field whose first character is `=`, `+`, `-`, `@`, TAB, or CR is prefixed with a single apostrophe *before* RFC-4180 quoting (`GRDBClipboardStore.swift:679-693`), the standard OWASP neutralization. | 7.2 | **Sensitive clips in exports** (audit 01 F-3.2): **fixed in this pass** across all three surfaces. Residual: the archive directory has no permissions tightening (`0700`) and no at-rest protection class on iOS temp dirs. | 
| 7.4 | **Restore path hardening is good**: version ceiling (`:109-111`), checksum-all-before-write, single transaction with rollback (`:140-161`), id-collision regeneration (`:153-155`). ## 8. iCloud/CloudKit — encrypted vs plain field exposure

**File:** `Packages/GanchoKit/Sources/GanchoSync/ClipRecordMapper.swift:41-77`

`contentText`) ride `record.encryptedValues` (`:60-66`); binary payloads ride `CKAsset`

`isSensitive`, `tags`, `boardIDs`, `sourceAppBundleID`, `sourceDeviceName` (`:41-57`).

| # | Finding | Risk/Impact | Fix | Sev | Eff |
|---|---|---|---|---|---|
| 8.1 | **`tags` ride PLAIN** (`:52-54`). Tags are user-authored text — "visa-card", "work-passwords", "divorce-lawyer" — content by any privacy-maximalist reading, and they're server-readable metadata today. Same argument as audit 01's `sourceDeviceName` note but stronger: tags are *typed by the user about the content*. | Server-visible user semantics | Move `tags` (and `sourceDeviceName`) to `encryptedValues`. | 8.2 | **`contentHash` plain** is required for dedupe/conflict, but it's SHA-256 of the content: anyone with a candidate plaintext can confirm the user copied it (confirmation-of-content oracle for the server). | 8.3 | **`isSensitive` plain** leaks *that* a secret was copied (not what). Low value to an attacker; needed server-side for nothing. Could move to encryptedValues with the same fallback dance as 8.1. | Minimal metadata leak | Bundle with 8.1's migration. | P3 | S |
