## Part 1 — Performance

### 1.1 The index inventory vs. every hot query

Indexes that exist (from the migrator, `Packages/GanchoKit/Sources/GanchoKit/GRDBClipboardStore.swift:192-374`):
`clip.createdAt` (v1, :197), `clip.contentHash` (v1, :203), `clip.pinboardID` (v4, :243, legacy),
`clip_board.clipID` + `clip_board.boardID` (v10, :310-313) plus the junction PK,
`purge_log.runAt` (v3), `mcp_access_log.occurredAt` (v9), and the FTS5 external-content table (v2).

Columns that appear in `WHERE`/`ORDER BY` with **no** supporting index: `isArchived`, `isPinned`,
`lastUsedAt`, `isSensitive`, `expiresAt`, `kind`, `sourceAppBundleID`, `isSnippet`, `keyword`,
`needsUpload`, `syncSystemFields`, `uses`. That drives the findings below.

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-1.1 | **P1** | S | **The most-executed query in the app cannot use any index.** `items(offset:limit:)` (`GRDBClipboardStore.swift:496-513`) filters `isArchived = 0` and orders by `isPinned DESC, IFNULL(lastUsedAt, createdAt) DESC`. No index covers either the filter or the expression order — SQLite full-scans and sorts the entire `clip` table for every page. `AppModel.refreshRecents()` (`Apps/GanchoMac/AppModel.swift:384-391`) runs it after **every capture, paste, pin, board change, undo, and sync settle** — i.e. tens of times per minute in normal use. At the documented 100k-row target this is a repeated 100k-row sort on the capture path, directly against the "<50 ms exact search" budget's spirit. **Fix (migration v16):** `CREATE INDEX clip_activity ON clip (isPinned DESC, IFNULL(lastUsedAt, createdAt) DESC) WHERE isArchived = 0` — SQLite supports partial + expression indexes; the expression must textually match what GRDB emits for `(Column("lastUsedAt") ?? Column("createdAt")).desc` (GRDB emits `IFNULL`). Add an `EXPLAIN QUERY PLAN` assertion to `PerformanceHarnessTests` so a future query edit can't silently de-index it. |
| A3-1.2 | **P1** | S | Same class of problem for `recentForBrowse(offset:limit:)` (`GRDBClipboardStore.swift:520-529`, `ORDER BY isPinned DESC, createdAt DESC`): the lone `createdAt` index can't serve a sort led by `isPinned`. This query backs the panel's grouped history (`Apps/GanchoMac/PanelView.swift:1218`), the iOS recent list (`Apps/GanchoiOS/GanchoiOSApp.swift:608`), **and the keyboard extension** (`Apps/GanchoKeyboard/KeyboardModel.swift:84`) — the last one inside a hard memory/CPU budget where a 100k-row sort per open is the difference between instant and janky. **Fix:** composite partial index `ON clip (isPinned DESC, createdAt DESC) WHERE isArchived = 0` in the same v16 migration. |
| A3-1.3 | P2 | S | **Counter queries full-scan on interactive paths.** `pinnedCount()` (`Pinboards.swift:65-69`) runs on every pin toggle (`AppModel.swift:976`); `sensitiveCount()` (`GRDBClipboardStore.swift:562-569`), `archivedCount()` (`TierEnforcement.swift:67-71`), `snippetCount()` (`SnippetLibrary.swift:51`) run on Privacy Center / paywall opens. Each is `COUNT(*)` over an unindexed flag → full scan. **Fix:** tiny partial indexes (`WHERE isPinned = 1`, `WHERE isSensitive = 1 AND isArchived = 0`, `WHERE isSnippet = 1`, `WHERE isArchived = 1`) turn each into an O(matches) index count. One migration, four lines. |
| A3-1.4 | P2 | S | **Retention purge cost at 100k rows** (`RetentionEngine.swift:24-70`). Good news first: `id NOT IN (SELECT clipID FROM clip_board)` is **uncorrelated** — SQLite materializes the subquery once per statement (and `clipID` is `NOT NULL`, so the `NOT IN` NULL-trap doesn't apply). The real cost is that each of the 4+K `DELETE` statements full-scans `clip` (no index on `expiresAt`, on `isSensitive+createdAt`, or on `kind+createdAt`), fires the FTS5 sync triggers per deleted row, and the whole pass reruns **every 300 s** from `AppModel.scheduleRetention()` (`AppModel.swift:1227-1233`) plus once per launch. **Fix:** (a) partial index `ON clip (expiresAt) WHERE expiresAt IS NOT NULL` and `ON clip (createdAt) WHERE isPinned = 0 AND isSnippet = 0`; (b) a cheap pre-check (`SELECT EXISTS(... createdAt <= ? OR expiresAt <= ?)`) to skip the pass entirely when nothing can expire — the common case for the 5-minute timer; (c) prefer `NOT EXISTS (SELECT 1 FROM clip_board WHERE clipID = clip.id)` and verify with `EXPLAIN` that the junction index is used for the anti-join. (Behavioral question about board members never expiring is F-4.2 — not repeated here.) |
| A3-1.5 | P2 | S | `RetentionEngine.runPurge` → `removeOrphanedBlobs()` (`RetentionEngine.swift:72`, `:81-89`) lists the **whole blobs directory** (`BlobStore.removeAll(except:)`, `BlobStore.swift:157-169`) and fetches the full distinct-hash set on **every** 5-minute pass, even when the purge deleted zero rows. **Fix:** skip the sweep when `summary.totalRowsPurged == 0`; consider running it hourly/daily rather than per purge. |
| A3-1.6 | P3 | S | `reformatLegacyImagePreviews` (`LegacyPreviewBackfill.swift:14-27`) runs a `LIKE 'Image (% bytes)'` scan of the table inside a **write transaction on every store open** (`GRDBClipboardStore.swift:69`). Idempotent by data-shape, but it still pays a scan + write-lock at every launch of the app, CLI, and each iOS extension that opens the store. **Fix:** gate behind a one-time marker (a `pragma user_version`-style row or defaults flag), or fold it into a proper migration. |

### 1.2 Sync scaling: the adapter loads what it only needs to count

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-1.7 | **P1** | M | **`pendingUploads()` hydrates full clip content — blobs read from disk and AES-decrypted — for every pending row, and three of its four callers only need IDs or a count.** The implementation (`SyncLocalStore.swift:68-83`) does a per-row `content(for:)` in a sequential `await` loop (N+1: one read txn + one blob read per row). Callers: `emitCurrentStatus` uses **`.count` only** (`CKSyncEngineAdapter.swift:161-165`, and it runs after every fetch/send event); `reenqueuePendingWork` needs **IDs only** (`:184`); `reconcilePendingChanges` needs **IDs only** (`:214`); only `nextRecordZoneChangeBatch` (`:242`) needs content. First sync of a 10k-clip history with images = 10k blob decryptions just to *emit a status line*. **Fix:** add `pendingUploadCount()` and `pendingUploadIDs()` to `SyncLocalStore` (both single indexed queries once A3-1.8's index exists) and keep content hydration only in the batch provider. |
| A3-1.8 | P2 | S | `WHERE syncSystemFields IS NULL OR needsUpload = 1` (`SyncLocalStore.swift:74`) has no index — full scan per call, several calls per sync cycle. **Fix:** partial index `ON clip (createdAt) WHERE syncSystemFields IS NULL OR needsUpload = 1` (SQLite allows the disjunction in a partial-index WHERE). |
| A3-1.9 | **P1** | M | **`nextRecordZoneChangeBatch` builds a `CKRecord` (with `CKAsset` files and full content) for *every* pending upload up front** (`CKSyncEngineAdapter.swift:241-267`), then hands `CKSyncEngine` a dictionary — but the engine batches sends (~a few hundred records); on an initial upload of a large history the entire library is resident as CKRecords at once, plus two extra store round-trips per row (`systemFields(for:)`, `boardIDs(forClip:)` — `:244-245`). **Fix:** build records only for the IDs present in `context`-relevant `pendingChanges` (the parameter already scopes them), cap the map at the batch size, and batch the two per-row lookups into single queries. |
| A3-1.10 | P2 | M | **Swallowed errors in the remote-apply path can permanently drop records.** Every `store.applyRemoteUpsert/...Deletion` in `handleFetchedRecordZoneChanges` is `try?` (`CKSyncEngineAdapter.swift:339-357`, 35 `try?` in the file). If a write fails (disk full, encryption key unavailable during a lock window), the event completes, `stateUpdate` persists the advanced change token (`:272-275`), and the record **never re-arrives** — the exact failure mode `resetSyncAndRepull` exists to repair, silently self-inflicted. **Fix:** collect failures per event; on any failure emit `.failed`, record a content-free `DiagnosticLog` entry, and skip persisting that event's state serialization so the batch is re-fetched. This is cheap and turns a silent data loss into a retried sync. |

### 1.3 Semantic search: the vectorized engine exists but the shipping path doesn't use it

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-1.11 | P2 | S | `semanticSearch` (`SemanticSearch.swift:39-53`) decodes each stored BLOB into a fresh `[Float]` (`Array($0.bindMemory…)` — one allocation + copy per row), then computes cosine with a **scalar zip loop**, recomputing the stored vector's norm on every query. `EmbeddingIndex` (`GanchoAI/EmbeddingIndex.swift:79-117`) already does this right: unit-normalize on insert, contiguous flat storage, `vDSP_dotpr` per row. At the documented 10k×512 budget the scalar path is ~5–10× the vectorized one and allocates ~10k arrays per keystroke-triggered query (`ClipboardQA`, board suggestion, search). **Fix, staged:** (a) S: normalize in `saveEmbedding` and score with `vDSP_dotpr` over `withUnsafeBytes` directly on the BLOB `Data` (no `[Float]` materialization; add a one-time re-normalize migration for existing vectors); (b) M: keep a process-lifetime `EmbeddingIndex` hydrated once per launch and appended on `saveEmbedding` — embeddings are device-local (never synced, per `GanchoiOSApp.swift:895`), so invalidation is only local inserts/deletes. |
| A3-1.12 | P2 | S | **Embedder/annotator instances are constructed per call at the app edge.** `ContextualSentenceEmbedder()` is built inline before every embed (`AppModel.swift:627, 1064; GanchoiOSApp.swift:513, 914`), `TieredClipAnnotator()` per enrichment (`AppModel.swift:362`), `ImageTextExtractor()` per OCR. `NLContextualEmbedding` model loading is the expensive part; a fresh instance per call risks paying it repeatedly and defeats the "caches its model after the first call" comment (`AppModel.swift:373-374`), which is only true per instance. **Fix:** one `EnrichmentService` (actor) owning the embedder/annotator/extractor for the process; also the natural seam for A3-2.4's dedup. |

### 1.4 Export, pagination, memory over long sessions

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-1.13 | P2 | M | **`exportJSON`/`exportCSV` materialize the entire table — full `contentText` included — in memory, twice** (rows array + encoded `Data`), with `.prettyPrinted, .sortedKeys` on top (`GRDBClipboardStore.swift:630-659`). At 100k clips this is easily hundreds of MB peak and a multi-second stall; on iOS it risks a jetsam right when the user asks for a backup. **Fix:** stream — `ClipRow.fetchCursor` + incremental `FileHandle` writes (both formats are line/element-append-friendly), return a file URL instead of `Data`. Note the protocol (`ClipboardStore.swift:25-26`) returning `Data` bakes the wrong shape into the frozen contract (see A3-2.6) — change it to a URL-writing API before freezing. (Sensitive-by-default and CSV formula injection are F-3.2/F-3.3.) |
| A3-1.14 | P2 | M | **OFFSET pagination is O(offset) and shift-unstable.** `items`/`recentForBrowse` page with `LIMIT ? OFFSET ?`; scrolling deep into 100k rows re-scans everything before the page, and a capture landing between two page loads shifts rows so the next page can duplicate/skip one (the guards at `PanelView.swift:1238` / `GanchoiOSApp.swift:624` only protect against a *changed list length*, not a same-length shift). **Fix:** keyset pagination — `WHERE (isPinned, createdAt, id) < (?, ?, ?) ORDER BY … LIMIT ?` keyed off the last row of the previous page; pairs with the A3-1.2 index so each page is an index seek. |
| A3-1.15 | P2 | S | **Both app-side thumbnail caches are unbounded — a long-lived menu-bar agent leaks by design.** `Apps/GanchoMac/ClipThumbnailStore.swift:14` and `Apps/GanchoiOS/ClipThumbnailStore.swift:15` hold `cache: [UUID: Image]` with **no eviction ever** (grep confirms no `removeAll`/evict). Each entry is a decoded 480 px image (~0.9 MB bitmap); browsing an image-heavy history over a weeks-long agent session grows resident memory monotonically. The keyboard extension already has the right pattern — FIFO cap of 24 (`KeyboardModel.swift:37, 163-167`). **Fix:** apply the same cap (~64 on Mac/iOS) or key the storage in an `NSCache`; one shared implementation per A3-2.7. |
| A3-1.16 | P2 | M | **Cold launch does storage work on the main actor.** `AppModel.init` (`AppModel.swift:162-292`) synchronously opens the encrypted store: Keychain read, SQLCipher key, **all pending migrations** (`GRDBClipboardStore.swift:68`), the blob-encryption marker check, and the A3-1.6 backfill — before the first frame. Normally fast, but the first launch after an update that ships a data migration (or a v15-style `UPDATE … WHERE id IN (SELECT …)` over 100k rows) freezes the app at the dock bounce, and on iOS risks the watchdog. **Fix:** an async store factory (`Task` off the main actor) with `AppModel` starting on a lightweight "opening…" state; the in-memory-fallback path already proves the model can run storeless. |
| A3-1.17 | P3 | — | **Capture loop re-validated at depth — sound** (confirms F-2.3). Policy math is pure and tested (`AdaptivePollingPolicy.swift`); `tick()` does only a `changeCount` read per turn (`MacPasteboardMonitor.swift:195-197`); vetoes are metadata-only and precede any read (`:207-234`); the read is detached at `.utility` (`:297-304`); bursts coalesce via `pendingRead?.cancel()` (`:253-258`) so no task chain builds up — intermediate burst copies are intentionally dropped (latest-wins), which is correct for a pasteboard that only exposes its latest content. One residual nit: `readPayload()` itself isn't cancellable mid-read (documented at `:294-296`) — under the "Ask" permission a stale ~2 s read still completes and is discarded; acceptable. No change. |
| A3-1.18 | P3 | — | **Keyboard extension memory ceiling re-validated** (confirms F-2.5): thumbnails FIFO-capped, decodes the store's small cached thumbnail via `thumbnailData(for:)` (never the original blob — `KeyboardModel.swift:159`, backed by `BlobStore.swift:84-94`), sensitive clips excluded. The one improvement for it is A3-1.2's index (its `recentForBrowse(limit: 60)` currently sorts the whole table in-process budget). |

## Part 2 — Architecture & Refactoring

### 2.1 Module boundaries vs. `docs/ARCHITECTURE.md` — verified, one structural bend

**Held:** CloudKit imports exist only in `GanchoSync` (grep: `ClipRecordMapper.swift`,
`CKSyncEngineAdapter.swift`, `BoardRecordMapper.swift` — nothing else in `Sources/` or `Apps/`
imports it; the apps get entitlement checks via `GanchoSync`'s non-CloudKit surface).
Telemetry networking is confined to `GanchoTelemetry` → TelemetryDeck (`Package.swift:66-70`);
no `URLSession` in the engine room (`LicenseActivationService` takes an injected transport —
`AppModel.swift:959`). The layering diagram (`docs/ARCHITECTURE.md:36-60`) matches
`Package.swift` exactly. Apps do not link `GanchoMCP` (per `Package.swift:28-31`).

**Bent:**

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-2.1 | **P1** | M | **The `ClipboardStore` protocol is not the app's real dependency — the concrete class is.** `GanchoiOSApp.swift` downcasts `store as? GRDBClipboardStore` **22 times**; `AppModel` keeps a parallel `grdbStore` handle (`AppModel.swift:48`); `KeyboardModel` types the field as the concrete class (`KeyboardModel.swift:45`). Everything interesting — boards, snippets, search, semantic search, sync counts, tier archiving — lives only on `GRDBClipboardStore` extensions, so the protocol boundary the architecture doc promises ("future non-Apple transports/clients behind the boundary") exists on paper but every feature bypasses it. This is also why the in-memory fallback silently loses boards/search: the casts return nil and features no-op. **Fix:** split the store surface into small protocols — `ClipSearching` (FTS/regex/semantic), `BoardStoring`, `SnippetStoring`, `ThumbnailProviding`, `RetentionMaintaining` — plus the existing `ClipboardStore`, `SyncLocalStore`, `MCPClipStore`; define `typealias FullClipStore = ClipboardStore & ClipSearching & …`; have `GRDBClipboardStore` conform (it already implements everything) and the apps hold `any FullClipStore`. Zero behavior change, kills all 22 casts, and makes A3-2.6's client contract real. |

### 2.3 `GRDBClipboardStore` cohesion

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-2.6 | P2 | M | The store is one class faceted across **nine same-module extension files** (semantic, MCP, snippets, backfill, smart collections, retention, pinboards, tier, sync — grep at top of this audit) sharing the `internal let writer` (`GRDBClipboardStore.swift:18`). Verdict: **keep the single class** — one serialized `DatabaseWriter` shared by capture/retention/sync is the concurrency story (validated as F-4.3), and splitting it into multiple writer-owning objects would *create* races. What should move out: (a) **export** — CSV/JSON formatting is not storage; a `ClipExporter` type over a row cursor is also where the F-3.2/F-3.3 fixes and A3-1.13 streaming land naturally; (b) the **public API** should be fronted by the A3-2.1 protocol facets so the *class* stops being the contract. Then mark GRDB-shaped members (`writer`, `migrate(upTo:)`, `thumbnailURL`'s plaintext-only semantics — `GRDBClipboardStore.swift:614-624`) `internal`/`@_spi(GanchoInternal)`. **Client contract to freeze (extends F-1.5):** `ClipItem`/`ClipContent`/`ClipSearchQuery`/`Pinboard` + the protocol facets + `GanchoArchive` + `SyncEngine` + `TextEmbedding` + `MCPClipStore`. Everything else is implementation. |
| A3-2.7 | P2 | M | **Thumbnail/downsample logic exists four times:** `Apps/GanchoMac/ClipThumbnailStore.swift:49-59`, `Apps/GanchoiOS/ClipThumbnailStore.swift:45-60`, `Apps/GanchoKeyboard/KeyboardModel.swift:173-188`, and `BlobStore.makeThumbnailData` (`BlobStore.swift:228-249`) — same ImageIO incantation, four copies, three cache policies (unbounded, unbounded, FIFO-24). Extends F-1.3 with the full inventory. **Fix:** one `ImageDownsampler` in GanchoKit (pure `Data → Data`, platform-free) + one generic capped `ThumbnailCache` in `GanchoDesign`/`GanchoAppCore` with a platform `Image` bridge at the edge. Note the two app stores also *load different bytes*: macOS reads the **full blob** per thumbnail (`AppModel.swift:172-177`) while iOS/keyboard read the small cached thumbnail via `store.thumbnailData(for:)` — after unification, macOS should switch to `thumbnailData` too (cheaper and correct for encrypted stores). |
| A3-2.8 | P3 | S | `IOSAppModel` reaches `GanchoSync` for `CloudKitEntitlements`/`SyncEnablement`/`SyncEngineFactory` — allowed (GanchoSync is the sanctioned edge), and `NoopSyncEngine` keeps free-tier paths network-free. Confirmed no violation; noted because `SyncController` (A3-2.4) becomes the single place this wiring lives. |

## Part 3 — Maintainability

| ID | Sev | Eff | Finding |
| --- | --- | --- | --- |
| A3-3.1 | P3 | — | **Swift 6 / `@unchecked Sendable` audit — all production uses justified.** Verified each: `UncheckedSendableBox` (read-only handoff, `CKSyncEngineAdapter.swift:424-428`); `TelemetryPipeline` (every member access under `NSLock`, `Telemetry.swift:85-113`); `DiagnosticLog` (locked ring buffer, `DiagnosticLog.swift:8-40`); `InMemoryPrivacyEventRecorder` (locked, `PrivacyEvents.swift:65-100`); `LicenseActivationService` (immutable, documented at `LicenseActivation.swift:82-85`); `InMemoryLicenseTokenStore` (locked test double). Remaining uses are in tests. Optional cleanup: the three locked classes are textbook `Mutex<State>` (Synchronization framework) candidates, which would drop the annotation entirely. |
| A3-3.2 | P2 | M | **`try?` swallowing is systemic at the edges: 71 in `AppModel.swift`, 59 in `GanchoiOSApp.swift`, 35 in `CKSyncEngineAdapter.swift`.** Two tiers of severity: (a) sync-apply swallowing loses data (A3-1.10 — fix first); (b) user-action swallowing (`togglePin`, `assign`, `promoteToSnippet`, board CRUD) shows success toasts *before/regardless of* whether the write landed (`AppModel.swift:982-983` shows the toast unconditionally after `try?`). Content-free by construction, consistent with the no-content-logging invariant. |
